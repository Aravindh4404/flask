# Flask Codebase Knowledge Graph

Generated by static analysis + code reading of `src/flask/`.  
LSP (Python) was unavailable; relationships derived from imports, class hierarchies, and call-site reading.

## 1. Module Dependency Map

### Notation
```
A → imports → B          (A depends on B at runtime)
A → type-only → B        (A imports B inside TYPE_CHECKING only)
A → extends → B          (class inheritance)
A → calls → B.func()     (direct call to function or method)
A → emits → Signal       (sends a blinker signal)
Signal → consumed by → A (A connects to the signal)
```

## 3. Request Lifecycle Call Chain

```
WSGI server
  → Flask.__call__(environ, start_response)
      → Flask.wsgi_app(environ, start_response)
          → AppContext.from_environ(app, environ)
              → app.request_class(environ)        [creates Request]
          → AppContext.push()
              → sets globals._cv_app ContextVar
              → emits signals.appcontext_pushed
              → AppContext.match_request()
                  → url_adapter.match()           [werkzeug routing]
                  → stores url_rule, view_args on Request
          → Flask.full_dispatch_request(ctx)
              → emits signals.request_started
              → Flask.preprocess_request(ctx)
                  → runs url_value_preprocessors (per scope)
                  → runs before_request_funcs (per scope)
              → Flask.dispatch_request(ctx)
                  → app.ensure_sync(view_functions[rule.endpoint])(**view_args)
                        [calls the view function / MethodView.dispatch_request]
              → Flask.finalize_request(ctx, rv)
                  → Flask.make_response(rv)
                      → Flask.ensure_sync / response_class(...)
                  → Flask.process_response(ctx, response)
                      → runs after_request_funcs (per scope, reverse)
                      → ctx._after_request_functions (after_this_request)
                      → app.session_interface.save_session()
                  → emits signals.request_finished
          → AppContext.pop(error)
              → Flask.do_teardown_request(ctx)
                  → runs teardown_request_funcs
                  → emits signals.request_tearing_down
              → Flask.do_teardown_appcontext(ctx)
                  → runs teardown_appcontext_funcs
                  → emits signals.appcontext_tearing_down
              → restores globals._cv_app to prior token
              → emits signals.appcontext_popped
```

## 4. Template Rendering Call Chain

```
render_template(name, **ctx)              (templating.py)
  → app_ctx._get_current_object()         (reads globals._cv_app)
  → ctx.app.jinja_env.get_or_select_template(name)
      → DispatchingJinjaLoader.get_source()
          → app.jinja_loader (FileSystemLoader for app templates)
          → blueprint.jinja_loader (for each registered blueprint)
  → templating._render(ctx, template, context)
      → app.update_template_context(ctx, context)
          → runs template_context_processors (per scope)
      → emits signals.before_render_template
      → template.render(context)          [Jinja2]
      → emits signals.template_rendered
```

## 5. Blueprint Registration Flow

```
Flask.register_blueprint(blueprint, **options)   (sansio/app.py)
  → Blueprint.make_setup_state(app, options, first_registration)
      → returns BlueprintSetupState
  → blueprint.register(app, options)             (sansio/blueprints.py)

```

## 7. Impact Map

For each major component, what breaks or is materially affected if it changes.

---

### `typing.py`
**Direct impact:** None at runtime (TYPE_CHECKING-only). All type annotations across the
codebase fail mypy/pyright if type aliases here are changed.  
**Affected:** Every module that annotates with `ft.*` types — practically the entire source.

---

### `signals.py`
**Direct impact:** Renaming or removing a signal breaks every extension that calls
`signal.connect()`, plus the internal emitters in `app.py`, `ctx.py`, `templating.py`,
`helpers.py`.  
**Affected:** `app.py`, `ctx.py`, `templating.py`, `helpers.py`, and any user/extension code
that subscribes to signals.

---

### `globals.py` / `_cv_app`
**Critical path.** Every proxy (`current_app`, `g`, `request`, `session`) is backed by
`_cv_app`. Changing the `ContextVar` name, type, or structure breaks:
- `app.py` — all `ctx`-aware methods read from `app_ctx`
- `ctx.py` — `AppContext.push/pop` sets/restores `_cv_app`
- `helpers.py`, `views.py`, `templating.py`, `json/__init__.py`, `wrappers.py`, `cli.py` — all
  dereference a proxy at call time  
**Affected:** Everything that runs inside a request or app context.

---

### `config.py`
**Direct impact:** `Config` is the dict that backs `app.config`. `ConfigAttribute` is used as
a descriptor on `App` for shorthand properties (`DEBUG`, `TESTING` etc.).  
- Changing `Config` key semantics breaks `sansio/app.py` defaults and every feature that reads a
  config value (`sessions.py`, `wrappers.py`, `helpers.py`, `blueprints.py`).  
- Removing `ConfigAttribute` breaks the descriptor properties on `App`.

---

### `logging.py` / `create_logger`
**Direct impact:** Called once in `App.__init__`. If `create_logger` signature or return type
changes, the `app.logger` property on `sansio/app.py` breaks.  
**Affected:** `sansio/app.py`, any code that calls `app.logger` (all Flask-internal error handling).

---

### `json/tag.py`
**Direct impact:** Used exclusively by `SecureCookieSessionInterface` to serialise session data.  
- Adding/removing a tag silently changes the wire format of session cookies — **existing
  sessions become unreadable if tags are removed**.  
**Affected:** `sessions.py`, production session cookies.

---

### `json/provider.py` / `JSONProvider`
**Direct impact:** `app.json` is an instance of this. Every call to `flask.json.dumps/loads` or
`jsonify` delegates here at runtime.  
- Changing the abstract interface (`dumps`, `loads`, `response`) breaks `DefaultJSONProvider`
  and any user-defined provider.  
- Changing `DefaultJSONProvider` changes serialisation of all JSON responses including
  `jsonify` and the session cookie (indirectly via `TaggedJSONSerializer`).  
**Affected:** `json/__init__.py`, `wrappers.py` (json_module), `sessions.py`.

---

### `wrappers.py` (`Request`, `Response`)
**Direct impact:** `Flask.request_class` and `Flask.response_class` default to these.
- Changing `Request` affects URL rule matching, blueprint detection, form data loading.
- Changing `Response` affects `make_response`, session cookie writing, `jsonify`.  
**Affected:** `app.py`, `ctx.py`, `sessions.py`, all view functions, testing.

---

### `sessions.py` (`SessionInterface`, `SecureCookieSessionInterface`)
**Direct impact:** `Flask.session_interface` defaults to `SecureCookieSessionInterface()`.
- `open_session` is called from `AppContext._get_session()` on first session access per request.
- `save_session` is called in `Flask.process_response()`.  
- Changing the cookie signing breaks existing client sessions (itsdangerous key rotation aside).  
**Affected:** `ctx.py`, `app.py` (`process_response`), all views that use `session`.

---

### `sansio/scaffold.py` (`Scaffold`)
**Direct impact:** Base class for both `App` and `Blueprint`. It owns all the hook registries
(`before_request_funcs`, `after_request_funcs`, etc.) and the `route`/`add_url_rule` API.  
- Changing the data structure of any hook registry (`defaultdict(list)`) breaks `sansio/app.py`'s
  `preprocess_request`, `process_response`, `do_teardown_*` which iterate over them.  
- Changing `setupmethod` changes when registration is allowed / disallowed.  
**Affected:** `sansio/app.py`, `sansio/blueprints.py`, and by extension `app.py`, `blueprints.py`.

---

### `sansio/blueprints.py` (`Blueprint`, `BlueprintSetupState`)
**Direct impact:** `BlueprintSetupState.add_url_rule` is the bridge from blueprint-relative
rules to `Flask.add_url_rule`. Changing prefix/subdomain resolution logic here changes how
all blueprint routes are mounted.  
**Affected:** `blueprints.py`, `app.py` (`register_blueprint`), every app that uses blueprints.

---

### `sansio/app.py` (`App`)
**Direct impact:** `App` owns the URL map (`werkzeug.routing.Map`), the blueprint registry,
the Jinja env, the JSON provider, the logger, and many config-derived properties.  
- Changes to `create_url_adapter`, `iter_blueprints`, `update_template_context`, or
  `add_url_rule` propagate to every request handled.  
**Affected:** `app.py` (Flask inherits), `templating.py` (uses `app.iter_blueprints`),
`ctx.py` (uses `app.create_url_adapter`).

---

### `ctx.py` (`AppContext`)
**Direct impact:** The unified context object pushed for every request and CLI command.
Holds `g`, `request`, `session`, `url_adapter`.
- `push()` / `pop()` must remain a matched pair; any change to teardown ordering here changes
  when `teardown_request_funcs` and `teardown_appcontext_funcs` run.  
- Merging request+app context (Flask 3.2) means `has_request` must be checked before accessing
  `ctx.request`; code that assumes context is always a request context will break.  
**Affected:** `app.py` (wsgi_app, full_dispatch_request), `globals.py` (proxies), `helpers.py`
(`after_this_request`, `copy_current_request_context`), `templating.py`.

---

### `templating.py` (`DispatchingJinjaLoader`, `Environment`)
**Direct impact:**
- `DispatchingJinjaLoader` iterates `app.iter_blueprints()` to resolve templates. Adding or
  removing blueprint Jinja loaders here changes template resolution order.
- `_default_template_ctx_processor` injects `g` and `request` into every template context.
  Removing it breaks templates that reference `{{ g }}` or `{{ request }}`.  
**Affected:** `sansio/scaffold.py` (registers the ctx processor), `sansio/app.py` (creates the
Jinja env), every template rendered by the app.

---

### `helpers.py`
**Direct impact:** `url_for`, `flash`, `abort`, `make_response`, `redirect`, `send_file` are
the bread-and-butter of view functions. Changes to their signatures or behaviour affect nearly
all application code.  
- `_split_blueprint_path` is used by `wrappers.Request.blueprint` and `blueprints`; changing it
  alters blueprint-name resolution.  
- `_CollectErrors` is used in `app.py` error handling; removing it changes how multiple URL
  rule errors are reported.  
**Affected:** `sansio/scaffold.py`, `app.py`, `wrappers.py`, `ctx.py`, `blueprints.py`, user views.

---

### `cli.py` (`AppGroup`, `ScriptInfo`, `FlaskGroup`)
**Direct impact:**
- `AppGroup` is the Click group attached to `Flask.cli` and `Blueprint.cli`.
- `ScriptInfo` is the object that discovers and loads the Flask app from env vars or the
  `--app` flag. Changes to `find_best_app` affect `flask run`, `flask shell`, and
  `flask routes`.  
**Affected:** `app.py`, `blueprints.py`, `testing.py` (`FlaskCliRunner`), the `flask` CLI entry point.

---

### `blueprints.py` (`Blueprint`)
**Direct impact:** Concrete Blueprint adds `cli`, `send_static_file`, `open_resource` over the
sans-IO base. Changes here affect blueprint-level static file serving and CLI command grouping.  
**Affected:** All applications that register blueprints and use `blueprint.cli` or
`blueprint.send_static_file`.

---

### `debughelpers.py`
**Direct impact:** Lazily imported only in debug mode. `attach_enctype_error_multidict` wraps
`request.files` to give better error messages for missing `enctype`; `explain_template_loading_attempts`
logs template loader search paths.  
- Safe to change without affecting production (only active when `app.debug = True`).  
**Affected:** `wrappers.Request._load_form_data()`, `templating.DispatchingJinjaLoader`.

---

### `testing.py` (`FlaskClient`, `FlaskCliRunner`)
**Direct impact:** `FlaskClient` wraps `werkzeug.test.Client` and adds session transaction support.
`FlaskCliRunner` wraps Click's `CliRunner` with `ScriptInfo`. Changes here affect test suites.
- Not imported by any runtime module (only type hints in `sansio/app.py`).  
**Affected:** All test suites that use `app.test_client()` or `app.test_cli_runner()`.

---

### `app.py` (`Flask`)
**Most impactful module.** Owns the entire request lifecycle:
`wsgi_app → push context → full_dispatch_request → preprocess → dispatch → finalize → teardown → pop`.  
- Changing method signatures for `dispatch_request`, `preprocess_request`, `process_response`,
  `handle_exception`, etc. breaks subclasses and extensions that override them.
- The new `ctx: AppContext` first-parameter convention (Flask 3.2+) is enforced in
  `__init_subclass__`; old-style overrides trigger deprecation warnings and wrapping.
- `ensure_sync` / `async_to_sync` control async view execution; changes affect async support.  
**Affected:** Literally everything — all views, all hooks, all signals, all extensions.
